# 第41-100章重启执行清单

> 适用范围：`第41章 - 第100章`
>
> 用途：在重启 `41-100` 写作时，强制把“真实 BCE 写作、字数 gate、章型校验、审校落盘、日志、Telegram、GitHub”逐项打勾，不再依赖口头承诺。

---

## 一、重启总原则

1. `41-100` 必须视为重启重写，不得沿用旧失败产物。
2. 不允许人工代写正文冒充 BCE 写作结果。
3. 任何一章未通过硬 gate，不得进入日志、Telegram、GitHub。
4. 每完成 `5` 章才允许做一次 GitHub 推送。
5. 若 BCE 写作失败，本章停止，不得“先放行再说”。
6. 第一幕后半段越接近 `80-100`，越不得放松章型、字数和终检标准。

## 十五、失败处置规则

### A. 单章失败

若任一章在以下任一步失败：

- BCE 写作 gate
- 一致性校对
- Codex 二审
- 终检

必须执行：

- [ ] 本章状态标记为 `BLOCKED`
- [ ] 不得继续本章后续步骤
- [ ] 不得用人工正文补位
- [ ] 记录失败原因
- [ ] 只允许重跑本章，不允许“跳到下一章以后再补”

### B. 批次失败

若某一批中出现任一章未通过，则：

- [ ] 本批不得推 GitHub
- [ ] 本批 Telegram 不得宣称“已全部完成”
- [ ] 必须先清掉失败章，再继续本批收尾

### C. 连续失败

若同一批出现两章以上失败：

- [ ] 停止继续写后续章
- [ ] 重新检查本批共享包
- [ ] 重新核对 `chapter_type`
- [ ] 重新核对 BCE 当前 provider 和模型配置

## 十六、41-100 重启时的强制提醒

以下行为一律禁止：

- [ ] 为了不断更，先人工补正文再补流程
- [ ] BCE 没写出来，却继续走审校
- [ ] 字数不达标仍然放行
- [ ] Telegram / GitHub 先发先推，后面再补质量
- [ ] 口头说“这章算关键章”，却不改 `chapter_types.json`
- [ ] 80-100 区间为了赶进度把关键章写成提纲章

重启 `41-100` 的目标不是“尽快补回来”，而是：

- **把第 40 章以前的质量线重新建立起来**
- **让第一幕完整落地，而不是只把大纲内容快速搬运成正文**
- **为后续 101-400 章建立真正可长期执行的强约束流程**
