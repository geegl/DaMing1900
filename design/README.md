# 结构化规划层（design）

`OUTLINE.md` 继续作为剧情与节拍的长文本真源。  
`design/` 则作为章节生产流程的结构化控制层，专门保存不适合散落在大纲正文中的执行元数据。

## 目录职责

- `chapter_types.json`
  `41-100` 章的结构化章节类型真源。后续 `build_context_pack.py` 会从这里读取 `chapter_type`，不再依赖人工临场判断。

- `chapter_policy.md`
  前 100 章的章节类型判定规则。用于约束什么情况必须标成 `key`，避免章型漂移。

- `restart_checklist_041_065.md`
  第 41-65 章重启时的执行清单。用于逐项核对写作 gate、审校、日志、Telegram 与 GitHub 放行，不再靠临时记忆。

- `restart_checklist_041_100.md`
  第 41-100 章重启时的完整执行清单。覆盖第一幕后半段直到第 100 章，并对 `80-100` 增加额外强化检查。

- `act2_architecture.md`
  第二幕 `101-280` 的阶段总结构，用于先锁总纲，再逐步细化 `101-140` 与后续阶段。

- `act2_setting_notes.md`
  第二幕专属设定草案，优先收纳海军改革、全球补给链、郑和遗脉、国内反扑机制与思想推进。

- `act2_detailed_outline_101_250.md`
  第二幕 `101-250` 的逐章细化执行镜像。它从 `OUTLINE.md` 的第二幕细纲中提取，方便在 `design/` 层集中查看和执行，不替代 `OUTLINE.md` 的剧情真源地位。

## 设计原则

1. `OUTLINE.md` 负责讲故事、讲节奏、讲长线推进。
2. `design/` 负责流程控制、执行元数据、强约束配置。
3. 结构化字段优先放在 `design/`，不要继续埋在不同 outline 段落里。
4. 逐章细化如果进入可执行状态，必须在 `design/` 保留一份执行镜像，避免只存在于 `OUTLINE.md` 某一段里。
5. 后续如果增加：
   - 章节风险等级
   - 本章指定模型
   - 强制出场人物
   - 钩子类型
   都优先收敛到 `design/`。

## 当前强约束

- `41-100` 每章都必须先有 `chapter_type`
- `chapter_type` 只能是：
  - `normal`
  - `key`
- `key` 章用于重大节点、重大揭示、重大关系变化或幕结构拐点
